# 一、项目概述（V2.0升级版）

## 1.1 项目核心目标（迭代升级）

在V1.0「语音交互+后台自动化代操作」解决老人“不敢点、点不到、记不住”数字鸿沟痛点的基础上，完成**产品形态与智能能力的本质升级**。依托Hermes独立容器化智能体架构与Claude长文本大模型能力，打造**专属、有记忆、有情感、可进化**的适老数字伴侣。不仅实现微信通话、助老打车等基础场景自动化，更通过用户习惯画像、方言交互、主动关怀、自主策略优化，解决高龄老人情感孤独、操作记忆衰退、使用适配难的深层痛点，实现工具属性到陪伴属性的升级，进一步夯实商业化与社会价值。

## 1.2 产品定位（全新定义）

国内首款**单用户单智能体实例**的适老化AI数字伴侣App。基于Hermes容器化独立架构，为每位高龄老人分配专属带记忆的AI智能体实例，融合上海方言情感交互、长期用户习惯记忆、自主进化优化、被动指令执行+主动智能关怀双重模式，兼顾工具实用性与情感陪伴属性，适配75岁以上高龄老人的认知、视觉、语言、行为习惯，同时满足子女远程监护、远程配置、政府智慧养老集采需求。

## 1.3 核心升级关键词

Hermes独立实例、容器化隔离、Agentic Workflow智能体工作流、Claude 3.5、长期用户记忆、习惯画像、主动关怀、上海方言交互、上下文多轮对话、按需弹性启停、自主进化优化、开发自动化、环境标准化、部署隔离化、VLA视觉反馈闭环、端云协同轻推理、子女远程同步配置

## 1.4 项目范围（V2.1进阶迭代新增范围）

### 1.4.1 保留V1.0+V2.0核心能力

- 微信语音/视频通话AI自动化代操作
- 62580000助老打车热线系统级直达呼叫
- 原子化权限引导、防误触、异常兜底、短信告警机制
- Apple风格适老化视觉设计、WCAG2.1 AA级无障碍标准
- 无障碍服务+VLM+AutoGLM基础自动化架构
- 专属Hermes智能体实例、用户习惯画像体系、双模方言交互系统
- 多轮上下文记忆、智能体自主进化工作流、云端弹性部署机制
- 开发者AI自动化运维体系

### 1.4.2 V2.1全新进阶新增能力

- **标准化AI开发环境护栏**：落地.agentrules项目开发规范，统一AI代码生成、交互逻辑、UI、安全底线，实现开发环境标准化
- **全流程开发自动化体系**：搭建Agent专属自动化脚本库，实现方言测试、UI合规校验、日志分析、策略迭代全自动化
- **Hermes实例精细化编排**：基于FastAPI+Docker SDK实现实例全生命周期自动化管理，新增pause/unpause毫秒级冷热启停优化
- **前端VLA视觉反馈闭环**：重构自动化执行逻辑，实现「意图解析-屏幕视觉校验-精准执行-结果反馈」全闭环，解决传统脚本盲操作问题
- **端侧隐私保护墙**：基于OpenCV实现截屏实时脱敏，自动屏蔽人脸、数字、隐私输入信息，保障数据传输安全
- **端云双层推理架构**：云端大模型负责复杂语义与策略决策，端侧轻模型负责本地状态实时识别，平衡响应速度与智能度
- **子女端远程同步配置能力**：新增SyncConfig统一配置接口，支持子女远程修改老人常用联系人、App白名单、关怀规则，降低老人使用门槛

## 1.5 项目核心进化逻辑

V1.0：**被动工具型**——用户下达明确指令，App机械执行操作，无记忆、无思考、无情感，仅解决“操作难”问题。

V2.0：**主动陪伴型**——专属AI智能体记住用户习惯、偏好、日常行为，理解用户隐性需求，支持闲聊陪伴，主动发起关怀提醒，可自主优化适配策略，解决“操作难+孤独感+适配差”多重痛点。

V2.1：**生产级可进化型**——基于Agentic Workflow打通开发、部署、运维、迭代全流程自动化，实现环境标准化、部署隔离化、迭代智能化，同时补齐隐私安全、视觉精准执行、远程配置能力，让专属智能体从“可用陪伴”升级为“稳定、安全、持续进化的专属数字伴侣”。

# 二、产品需求（PRD）完整版（V2.0+V2.1进阶）

## 2.1 目标用户画像升级

在原有用户画像基础上，补充情感需求与方言使用特征，适配V2.0陪伴属性，V2.1新增子女用户远程配置诉求：

维度

核心老年用户（75岁+）

付费子女用户（30-55岁）

新增特征

多数熟悉上海方言，普通话表达不流利；独居比例高，存在情感孤独；操作习惯固化，学习能力弱，需要长期适配；行为规律固定（固定作息、固定使用习惯）

除安全监护外，新增**情感陪伴诉求**，希望AI缓解老人独居孤独，通过行为报告掌握老人日常状态；新增**远程配置诉求**，希望远程简化老人设备配置，降低老人操作负担

新增痛点

标准普通话交互不适应；单一工具功能无温度；设备操作反复出错，越用越焦虑；无人沟通、日常状态无人关注；设备配置复杂，无法自主完成初始化设置

无法实时了解老人日常行为状态；无法远程感知老人情绪与独居状态；无法远程帮老人配置常用功能、联系人，老人使用门槛高

新增诉求

方言交流、生活化闲聊陪伴、个性化操作适配、主动提醒关怀、极简初始化配置

获取老人每日行为画像报告、查看AI陪伴记录、远程自定义老人关怀规则、远程配置常用联系人与应用白名单

## 2.2 核心功能全新升级需求（含V2.1进阶）

### 2.2.1 核心能力升级：从指令执行到意图预测+主动关怀

V2.0彻底颠覆传统AI工具的被动交互模式，基于Hermes专属智能体的长期记忆能力，实现**被动执行+主动预判**双模式运行。V2.1进一步优化执行逻辑，通过VLA视觉闭环解决传统自动化脚本盲操作、适配性差的问题，实现精准可视化执行。智能体持续学习老人的作息习惯、操作偏好、语言习惯、行为规律，形成专属用户习惯画像。

**核心场景示例：**

- 老人日常固定早8点查看天气，若连续2天未操作，Hermes智能体主动用上海方言询问：“阿叔，今朝哪能没看天气啦？身体还好伐？”
- 老人常用微信视频联系女儿、语音联系儿子，智能体自动记忆偏好，老人模糊说“打个电话给囡恩”，自动匹配视频通话
- 老人多次调整播报音量，智能体记忆常用音量档位，后续默认适配用户偏好

### 2.2.2 双模交互体系（方言+双语、指令+闲聊）

#### 1、语言能力：上海方言+普通话双模适配

- **方言识别**：精准识别上海本地口语、俚语、模糊表达，支持带口音普通话、纯上海话指令输入
- **方言播报**：TTS支持上海方言语音回复，语气亲切生活化，贴合老年用户语言习惯
- **自动适配**：智能体根据用户历史对话，自动默认用户常用语言，支持手动切换普通话/上海话

#### 2、交互模式：严格指令模式 + 闲聊陪伴模式

- **严格指令模式**：识别用户操作类指令（打电话、打车、打开医保码），优先执行手机自动化操作，响应高效无冗余对话
- **闲聊陪伴模式**：识别情感类、闲聊类语句（聊家常、问旧事、吐槽生活），启动情感对话，不执行设备操作，提供情绪陪伴
- **模式智能切换**：智能体自动语义判断，无需用户手动切换，无缝适配用户需求

### 2.2.3 多轮上下文记忆交互

基于Redis缓存短期对话上下文，结合Hermes数据库长期记忆，实现连贯多轮对话，解决传统AI单次对话无记忆的痛点。V2.1优化记忆策略，由Claude定期总结近50条对话生成摘要，精简存储、提升上下文匹配效率。

**场景示例：**

- 用户：“声音有点小”→AI调高播报音量并方言回复
- 用户：“再大点声”→AI识别上下文，继续调高当前音量，无需用户重复说明需求
- 用户：“帮我打视频给儿子”→执行操作→用户：“换成语音吧”→AI基于上下文修改操作指令

### 2.2.4 专属用户习惯画像系统

Hermes智能体每日凌晨自动执行画像分析任务，基于当日用户所有操作记录、对话记录、使用行为，生成**老人专属每日状态报告**，包含作息状态、设备使用情况、操作难点、情绪倾向、异常行为预警。V2.1新增习惯权重体系，量化用户不同时段、不同场景的操作偏好，提升主动预判精准度。

**画像核心维度：**

- 作息习惯：常用使用时段、固定操作行为、作息是否规律
- 操作偏好：常用功能、常用联系人、音量/字体偏好、操作习惯、场景化习惯权重
- 交互偏好：方言/普通话偏好、闲聊频率、对话风格偏好
- 行为异常：长期未使用高频功能、操作频繁失败、长时间无设备交互

报告可同步展示在子女端，支持子女查看、设置关怀阈值，触发异常主动提醒。V2.1新增子女远程配置同步能力，支持自定义老人使用配置。

### 2.2.5 Hermes智能体自主进化能力

基于Agentic Workflow工作流，智能体自动复盘用户操作日志，迭代个性化适配策略，实现**越用越适配、越用越智能**。V2.1依托标准化开发护栏与自动化脚本，实现策略迭代全自动化，无需人工干预。

**进化场景：**

- 若老人连续3次手动重试“打开医保码”操作，智能体自动优化该场景的自动化识别策略，降低操作失败率
- 若老人多次使用方言模糊指令，智能体自动更新本地语义库，适配用户专属口语习惯
- 自动更新`.agentrules`策略文件，沉淀用户专属辅助规则，无需人工迭代代码

### 2.2.6 V2.1进阶核心新增功能

#### 1、前端VLA视觉反馈闭环能力

重构传统自动化执行逻辑，摒弃固定坐标脚本模式，实现「AI意图解析→屏幕视觉脱敏采集→VLM坐标识别→精准执行→结果反馈重试」的全闭环自动化能力，彻底解决机型适配、界面更新导致的操作失效问题，大幅提升自动化成功率。

#### 2、端侧实时隐私保护墙

在MediaProjection截屏链路中嵌入OpenCV实时脱敏滤镜，自动识别屏幕内人脸、数字账号、手机号、身份证号等隐私区域，上传云端前完成涂黑脱敏，实现隐私数据端侧清零，从源头保障用户数据安全，符合养老数据合规要求。

#### 3、子女端SyncConfig远程同步配置

Hermes实例新增统一配置同步接口，支持子女Web端/小程序远程配置老人设备参数，核心可配置内容包含：常用通话联系人、App使用白名单、主动关怀提醒频次、方言/普通话默认模式、设备音量字体默认参数，配置实时同步至老人专属智能体与端侧设备，大幅降低老人初始配置门槛。

## 2.3 原有核心功能优化升级

### 2.3.1 自动化操作智能适配升级

### 2.3.2 权限保活与异常兜底升级

### 2.3.3 首页交互体验升级

- 闲置提示文字基于用户习惯个性化轮播，不再统一模板
- 主动关怀弹窗轻量化展示，不遮挡核心操作按钮，适配适老化防误触规则

## 2.4 商业化能力升级

### 2.4.1 C端订阅增值服务（新增）

- V2.1新增：子女远程设备配置、白名单管理、习惯偏好一键同步增值能力

### 2.4.2 G端集采升级

# 四、工程体系标准化与CI/CD落地规范（V2.1新增生产级工程体系）

## 4.2 全栈代码编码规范

### 4.2.1 Android客户端编码规范（MVI架构）

全局严格遵循\*\*MVI (Model-View-Intent)\*\*架构模式，统一分层逻辑，杜绝架构混乱：

- 状态管理：ViewState为唯一可观察UI状态，统一管理界面、语音、AI交互、自动化执行状态。
- 事件分发：ViewEvent统一接收用户长按、点击、唤醒等交互动作。
- 副作用处理：ViewEffect统一管控TTS播报、跨App跳转、接口请求等一次性操作。
- 依赖注入：所有外部服务（AI识别、语音、VLM、网络）均通过Hilt Module注入，禁止直接实例化。

**自动化执行强制规范**：

- 操作延时：所有自动化触控操作添加300ms-600ms随机延迟，模拟真人操作，规避微信安全拦截策略。
- 坐标校准：所有VLM返回坐标必须通过比例公式换算：Xreal = Xvlm × (设备宽度/标准宽度)，适配全分辨率设备。
- 截屏规范：MediaProjection截屏压缩至720p、70%质量、WebP格式，降低VLM推理时延与网络开销。

**适老化UI强制规范**：

- 固定核心色值：主色#FF9500（暖阳橙）、背景色#FFF8F0（暖米色）。
- 交互反馈：所有按钮点击触发长震感触觉反馈，适配老人感知能力。
- 字体规范：禁用px单位，统一使用sp，基准字号24sp，支持动态适配。

### 4.2.2 Hermes后端编码规范

#### 1、API设计规范

- 通讯协议：实时语音、指令流采用WSS WebSocket；数据查询、配置更新、习惯记录采用HTTPS REST。
- 统一错误码：1001（语音识别失败）、1002（意图路由冲突）、2001（自动化脚本超时）、4001（实例启动失败），全项目统一复用。

#### 2、提示词工程规范

- 固定系统角色Prompt，统一上海话交互风格、意图识别、容错、主动关怀规则。
- AI输出强制最小化JSON格式，精简Token消耗，提升传输与解析效率。

#### 3、数据库规范

- 用户隐私、习惯数据采用SQLCipher/Fernet加密存储，保障独居老人数据安全。
- 所有用户习惯写入、数据更新操作必须包裹数据库事务，防止容器异常导致数据损坏。

### 4.2.3 微信通话核心场景专属编码约束

针对跨App、带状态的高复杂度自动化场景，制定强制编码规则，保障核心功能稳定性：

- 前置状态校验：执行所有微信操作前，调用checkCurrentUI()校验当前界面，非微信界面则自动启动微信App。
- 后置视觉确认：点击视频通话后，通过VLM识别页面状态，校验“呼叫中”状态；识别对方忙、无人接听等异常，自动触发方言语音安抚提示。

### 4.6.2 Android端 CI/CD流水线（android-ci.yml）

触发规则：main、develop分支任意代码push触发，完成代码质检、单元测试、打包、制品上传全流程，杜绝代码改动导致自动化功能失效。

```
name: Android CI/CD (Hulao App)

on:
  push:
    branches: [ "main", "develop" ]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: set up JDK 17
        uses: actions/setup-java@v4
        with:
          java-version: '17'
          distribution: 'temurin'
          cache: gradle

      - name: Grant execute permission for gradlew
        run: chmod +x gradlew

      - name: Run Lint
        run: ./gradlew lintDebug

      - name: Run Unit Tests
        run: ./gradlew testDebugUnitTest

      - name: Build Release APK
        run: ./gradlew assembleRelease
        env:
          SIGNING_KEY_STORE_PASSWORD: ${{ secrets.SIGNING_KEY_STORE_PASSWORD }}
          SIGNING_KEY_ALIAS: ${{ secrets.SIGNING_KEY_ALIAS }}
          SIGNING_KEY_PASSWORD: ${{ secrets.SIGNING_KEY_PASSWORD }}

      - name: Upload APK
        uses: actions/upload-artifact@v4
        with:
          name: hulao-app-release
          path: app/build/outputs/apk/release/*.apk
```

这是一个非常关键的技术选型问题。简单直接的回答是：**Supabase 能够完美支持 Hermes 架构的“大脑”部分（数据、逻辑、检索），但无法直接运行 Hermes 的“身体”部分（即 Docker 容器实例）。**

为了让您清晰地理解如何将 Supabase 集成到您的架构中，我们需要从以下三个层面进行详细拆解：

### 1. Supabase 能承载的部分 (完全替代云服务器后端)

Supabase 可以直接负责您接口规范中定义的**管理层（Management Layer）**：

- **专属记忆库 (Memory DB)**：利用 Supabase 内置的 PostgreSQL。
  - **优势**：支持 `pgvector` 插件。您可以将老人的历史对话、行为习惯转化为向量存储。当老人说“打给小明”时，Hermes 可以通过向量检索快速找回“小明是老人的外孙”这一上下文。
- **Edge Functions (逻辑分发)**：替代传统的 FastAPI/Node.js 后端。
  - **场景**：当 Android 端发送意图时，Edge Function 负责逻辑鉴权、调用 Claude/AutoGLM 模型，并将解析后的指令包下发给 App。
- **Realtime (长连接控制)**：
  - **场景**：可以利用它的实时推送功能同步子女端对老人配置的修改（如子女在网页端新增了一个联系人，老人手机端实时生效）。

### 2. Supabase 无法直接支持的部分 (需要配合 SCNet 或其他资源)

由于您的 Hermes 架构采用了 **“一人一实例 (Docker)”** 的物理隔离方案，Supabase 的托管版本（Cloud）存在以下局限：

- **无法运行 Docker 容器**：Supabase 是一个 Serverless 平台，它不提供运行长期挂载的自定义 Docker 容器的服务。
- **计算能力限制**：Edge Functions 适合短时逻辑（如解析 JSON），但不适合运行高负载的 **VLM (视觉推理)** 这种重型模型。

### 3. 落地建议：Supabase + SCNet 混合架构

为了符合您的生产级清单要求，建议采取以下 **“强强联手”** 的配置：

#### A. 使用 Supabase 处理“元数据与状态”

- **用户表与鉴权**：管理老人和子女的绑定关系。
- **习惯画像表**：存储结构化的老人作息、高频操作。
- **Edge Functions**：作为 API 网关，处理 `X-Hermes-Token` 校验。

#### B. 使用 SCNet 处理“计算与隔离”

- **Docker 宿主机**：在 SCNet 申请的服务器上运行您的 `Hermes-Core` 容器集群。
- **VLM 推理**：利用超算中心的高性能算力处理屏幕截图。

### 4. 具体接口适配示例

如果您将后端切到 Supabase，您的接口调用逻辑将变为：

1. **前端发送请求**：Android App 调用 Supabase Edge Function。
2. **后端中转逻辑**：
   - Edge Function 从 Supabase DB 读取用户记忆。
   - Edge Function 调用云端 Claude API 获取意图。
   - **关键点**：如果需要复杂动作，Edge Function 发起一个内部请求给 **SCNet 上的控制中枢**，由中枢调度对应的 Docker 实例。
3. **结果返回**：Edge Function 组装最终指令包返回给 App。
