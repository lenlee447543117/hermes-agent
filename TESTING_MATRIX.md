## 三、 测试通过要求 (Final Acceptance Criteria)

### 1. 响应时延 (Latency) - 核心体验:
* **端到端响应：** 用户松手到辅助服务执行第一个动作，平均耗时应 ≤ 3.5s（取决于网络波动）。
* **本地处理：** WAV 封装与截图准备的总耗时应 ≤ 800ms。

### 2. 识别执行率 (Success Rate):
* **指令执行率：** 对于明确的视觉元素操作（如“点击搜索”），执行成功率需达到 **95%** 以上。
* **语义容错：** 当用户表述模糊（如“点那个绿色的”）时，AutoGLM 应能通过图像坐标正确锁定目标。

### 3. 系统鲁棒性 (Robustness):
* **服务常驻：** 辅助服务在后台运行时，不能被系统轻易杀掉（需检查 Notification 前台服务状态）。
* **状态恢复：** 若 API 返回 JSON 格式错误，App 需提示“没听清，请再说一次”，且不能中断 `AccessibilityService` 实例。

## 四、 故障排查清单 (Troubleshooting Checklist)

如果在测试中遇到问题，请按以下顺序排查：

1. **动作不执行？** 检查 `AutoGLMAccessibilityService.instance` 是否为空，确认用户是否在设置中手动关闭了开关。
2. **点击位置偏移？** 检查截图的 `Density`（像素密度）与辅助服务点击坐标是否进行了比例换算。
3. **录音全静音？** 查看日志中是否有 `ACDB Error`。若有，确认 `AudioSource` 是否已强制设为 `VOICE_COMMUNICATION`。
4. **接口返回 401？** 检查 `apiKey` 是否过期，以及 Base64 编码是否去掉了换行符 (`Base64.NO_WRAP`)。
